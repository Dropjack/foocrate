## 1. 如何使用这份 TODO

### 1.1 编号用于指挥，标题用于提交

每个三级标题中的两位数字只是路线编号，数字后的文字是唯一 Fork 提交标题。例如步骤 `00 整理项目规格与开发路线` 的提交标题是：

```text
整理项目规格与开发路线
```

不要把多个步骤压成一个提交，也不要把一个步骤拆成数个无法独立验收的提交。若实现中发现步骤过大或 SDK 迫使边界改变，必须先修改本 TODO、说明原因并让用户批准，不能悄悄改变提交范围。

### 1.2 用户可以这样指挥

- `开始 03 建立可安装的 x64 组件工程`：Codex 先检查所有前置步骤和 Git 状态，再开始该步骤。
- `继续当前步骤`：Codex 从 `tasks/README.md` 和本文件记录的位置继续，不重新做已完成工作。
- `给我验收步骤`：Codex 只给当前步骤的人工检查，不提前进入下一步。
- `验收通过`：Codex 更新规格、验证记录、任务状态和本 TODO，随后明确告诉用户使用哪个不带路线编号的标题提交。
- `我已经提交`：Codex 核对 HEAD 和工作树，确认提交确实对应当前步骤，再把下一项设为当前步骤。
- `这一步不通过：……`：Codex 保持当前步骤未完成，修复后重新构建、测试和提供验收，不让用户先提交。

### 1.3 每一步的固定工作流

1. Codex 严格读取启动文档、当前任务、本 TODO 和相关规格，检查并保护已有修改。
2. Codex 为本步骤建立或更新独立编号任务，写明中文程序逻辑、范围、失败行为和人工验收。
3. 涉及用户可见行为时，先完成参考观察和模块规格，请用户核对；未获批准不写功能代码。
4. Codex 实现完整纵向功能，更新设置迁移、错误处理和生命周期逻辑，不提交占位或假数据。
5. Codex 构建 x64 Debug 与 Release，运行自动测试，只部署到 `.local/foobar-dev` 或 `.local/foobar-test`。
6. 用户按本步骤的人工检查验收。自动检查通过只能称为“实现完成待验收”。
7. 用户明确验收通过后，Codex 同步状态和文档；用户在 Fork 中使用编号后的标题文字提交。
8. Codex 在用户确认已经提交后才推进下一项。
