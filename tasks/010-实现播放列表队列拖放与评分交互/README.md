## 任务目标

完成中央播放列表最后一组日常操作：查看和管理真实播放队列，在普通列表内拖动重排，从资源管理器和其他 foobar2000 面板拖入/拖出曲目，并直接点击列表五星评分。所有结果都必须落到 foobar2000 或 Playback Statistics 的真实数据中，不能只改变 Refrain 的显示。

## 明确不做

- 不创建 Foobox 的 `Queue Content` 临时播放列表；用户已接受这一边界。
- 不建立私有评分数据库，也不把评分写进音频文件标签。
- 不提前实现步骤 12 的 `Playlist Browser`；Refrain 内部可见的跨播放列表落点随左栏出现后接入。
- 不提前修改步骤 14–16 的按钮布局、右栏比例、字体、歌词或主题视觉。
- 不把自动/锁定列表伪装成可重排、可插入的普通列表。

## 中文程序逻辑

1. 输入来自活动播放列表、`playlist_manager` 的真实播放队列、Playback Statistics 的 `%rating%`、鼠标/键盘操作，以及系统 OLE 或 foobar2000 数据对象。
2. 程序只保存轻量界面状态：当前拖动类型、冻结的播放列表 GUID/项目快照、插入位置、队列快照和评分悬停值；播放列表、队列和评分仍由正式组件持有，队列面板使用共用响应式列比例且不增加专属持久化。
3. 用户开始拖动时先超过系统阈值，再判断来源、目标和锁定状态。内部移动用真实索引和 Undo；外部拖入异步解析；拖出由目标返回 Copy/Move 结果。不能证明成功时不删除来源。
4. 队列变化后重新读取真实顺序并刷新 `State / Queue` 列和队列面板。评分提交后等待 Playback Statistics 回调刷新列表、右上五星和自动播放列表。
5. 只持久化 `State / Queue` 列布局与队列面板几何信息；队列内容、拖动中状态和悬停评分不保存。
6. Esc、失焦、窗口关闭、列表/队列外部变化、异步结果晚到或目标失效都会取消未提交操作并释放结果；错误时恢复真实状态，不保留插入线或假成功显示。

## 验收修正阶段

3. 手动列表内部拖动只执行 Move，不能因同源 OLE 重入失败而复制；资源管理器仍可拖入可添加的手动列表，自动/锁定列表按 foobar2000 锁定过滤拒绝。
4. 外部拖入跳过目标中已有的精确曲目身份，但不清理用户原有重复项；跨播放列表名称与完整手动/智能列表管理随步骤 12 Playlist Browser 验收。

6. 修复播放期动态信息通知整批清空组图缓存造成的周期闪烁。

5. 本轮只修正上述回归；Playlist Browser、播放列表名称落点和自动/手动列表完整管理仍留到步骤 12。

第二轮修正结果：

- 右侧显示 Playback Queue 时，只有真正双击队列行才消费该事件；中央 Playlist View 的双击继续执行其真实曲目默认动作。
- 主列表鼠标按下时复制稳定的显示行身份，后续选择回调或自动折叠重建显示行不会破坏双击与拖动目标。
- 指针跨出 Playlist View 右边界即启动标准 OLE，不再要求额外越过 8 像素；同一 Refrain 面板生成的冻结选择直接被识别为可投放到 Playback Queue，不依赖外部文件格式二次探测。
- Group By 只组织显示；手动且未被 foobar2000 锁定的列表仍按真实列表索引自由重排。

## 第三轮验收反馈与根因

1. 已通过：右侧 Playback Queue 打开时，中央 Playlist View 双击播放恢复正常。
2. 仍未通过：Playlist View 拖入 Playback Queue；任何 Group By 下的新建普通手动列表也无法内部重排。
3. 用户提供的新建 `Untitled` 列表证明确实是普通手动列表，因此不再把失败归因于自动列表、锁定或分组权限。
4. 共同根因：内部拖动松手和启动 OLE 前都需要主动调用 `ReleaseCapture()`；Win32 会同步发送 `WM_CAPTURECHANGED`，现有代码把这次主动释放误判为异常丢失捕获并调用 `cancelInteraction()`，在真正提交前清空了曲目快照、选择与拖动状态。
5. 修正原则：主动释放只交还鼠标捕获并保留冻结快照，继续完成当前提交；只有窗口失焦、其他窗口抢走捕获或取消操作时才清空拖动状态。

第三轮修正已实现：通过内部保护标记区分主动 `ReleaseCapture()` 与真实异常捕获丢失；内部重排、拖入 Playback Queue 和队列自身重排在正常松手时都保留快照并执行提交。

### 人工验收清单

第二轮只需先复验以下三个阻断项：

1. 保持右侧 Playback Queue 打开，双击中央 Playlist View 中任意曲目；应立即播放该曲目。切回 Now Playing 后再双击一次，行为应相同。
2. 保持 Playback Queue 打开，把中央 Playlist View 中的单首及多首曲目拖进右侧队列；右栏出现接收反馈，松手后按选择顺序加入，来源列表不变，同一首可再次加入。
3. 对同一个普通手动列表依次启用 `Group by Directory`、`Artist / Album`、`Artist` 等分组，拖动单首和多首到组内及跨组位置；必须真实移动且总数不变。分组会按移动后的真实顺序重新生成，但不能禁止拖动。

以上三项通过后，再快速确认先前已经通过的队列双击、同源不复制和封面不闪烁没有回归。完整原始清单保留如下：

1. 点击底部右侧队列按钮，确认整个右栏在 `Now Playing Panel` 与 `Playback Queue Panel` 间切换；队列标题区不再有 Remove/Clear 按钮。右击队列曲目应有 `Play now`、`Remove selected`、`Clear playback queue` 和正式曲目菜单，Delete 仍可删除所选。
2. 用正式菜单把普通项目和同一项目多次加入队列；确认 `State / Queue` 列和队列顺序正确。拖动单项、连续及非连续多选，确认真实队列顺序改变；用下一曲按钮确认新顺序生效。
3. 双击或选中非首项后按 Enter：该项应立即播放并从队列中消费，其他队列项保持原相对顺序。
4. 从中央 Playlist View 把单项或多项拖到可见的 Playback Queue；确认按选择顺序加入、来源列表不变，而且同一曲目可以重复排队。
5. 在普通手动列表内拖动单项、连续/非连续多选，尤其复测“向列表顶部拖”的原问题：只能移动，曲目总数不变，不能复制出第二份；Esc 取消、Undo 恢复。自动/锁定列表及左侧 Playlist Browser 的完整人工验收留到步骤 12。
6. 从资源管理器向普通手动列表拖入文件或文件夹；再次拖入同一个精确曲目时应跳过重复并显示简短状态，不能产生新的列表重复项。测试失败或取消时不得留下半完成插入。
7. 保持包含组封面的列表播放至少 30 秒，并执行拖动进度、切歌和评分；组封面不能每秒闪烁或反复出现 Loading，F5 仍可主动刷新。随后回归列表 Rating 点击/清除、右上五星、分组、折叠和列配置。跨标准 Columns UI 播放列表面板的 OLE Copy/Move 可作为兼容性附加检查；Refrain 左侧播放列表名称落点属于步骤 12。
