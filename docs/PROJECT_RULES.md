# 项目总则

本项目实现一个 foobar2000 x64 组件，以 Columns UI SDK 和原生 Windows C++ 为基础，复现 Foobox 的主要布局、视觉层级和交互习惯。

Foobox 仅作为外观与可观察行为的参考，不继承其 JavaScript 宿主架构。本文件是项目长期协作规则；任何新任务开始前都必须先完整阅读本文件，再阅读 `tasks/README.md` 和当前任务文档。

## 沟通与可检查性

- 面向用户的说明、规格、任务、验收步骤和程序逻辑全部使用中文。
- C++ 标识符可以使用英文，但每个功能在写代码前必须有中文逻辑说明，使不熟悉 C++ 的用户能够理解数据从哪里来、经过什么判断、状态如何改变、界面如何响应以及失败时如何处理。
- 开始一项工作前，先用中文说明本轮目标、会查看或修改的范围以及明确不会做的事。
- 工作过程中持续报告当前正在做什么、已经确认什么、还有什么未确认；不得长时间静默后一次性交付大量不可核对的改动。
- 每个任务必须拆成可单独检查的步骤。每一步都要写明输入、动作、产物、检查方法和通过标准。
- 自动检查通过只能称为“实现完成，等待用户验收”。只有用户明确确认后，任务才能标记为“已验收”。
- 不假定用户理解 C++、Windows API、foobar2000 SDK 或构建系统；出现这些概念时必须同时给出人类语言解释。
- 不用“以后再补”“先占位”“先跑起来”等措辞掩盖缺失。若确有外部条件或技术风险导致本轮不能完成，必须明确记录原因、影响、证据和后续解除条件。

## 文本编码与启动协议

- 仓库内所有由项目维护的文本文件统一使用 UTF-8 无 BOM 和 LF 换行；第三方文件保持原样。
- 每次新会话或新任务启动时，第一步检查 `AGENTS.md`、`docs/PROJECT_RULES.md`、`tasks/README.md` 和当前任务文档是否为有效 UTF-8；不得依据乱码内容继续推断或假装已读懂。
- 在 PowerShell 中读取中文文本时，不依赖终端默认编码。需要验证时读取原始字节，并使用启用错误检测的 `System.Text.UTF8Encoding($false, $true)` 解码。
- 若发现 BOM、无效 UTF-8 或终端显示乱码，先修复读取/显示方式并向用户说明，再继续理解内容。未经确认不得把乱码文件直接重写。
- 新建或修改项目文本后，必须验证：可以严格按 UTF-8 解码、前三个字节不是 `EF BB BF`、中文内容未损坏。
- `.editorconfig` 是编辑器层面的编码与换行约束；本节是执行层面的强制检查，两者同时生效。

## 完整实现原则

- 项目不采用以删减需求换速度的 MVP 策略。每个进入实现的模块都以其已批准规格中的“可用状态”为目标。
- 不提交只有界面没有行为、只有成功路径、使用假数据、硬编码临时结果、空实现、无持久化的临时设置或无法验收的阶段性代码。
- 允许按纵向功能单元安排先后顺序，但每个单元一旦标记“实现完成”，就必须具备完整交互、错误处理、生命周期处理、设置持久化、自动检查和人工验收步骤。
- 可以使用开发脚手架、诊断日志和测试替身，但不得把它们冒充用户可用功能；它们必须有明确用途且不能污染 Release 行为。
- 参考项目完整存在，因此默认目标是还原已确认的行为。只有 SDK 限制、平台差异、许可证、安全性、可访问性或可证明的技术冲突，才可以提出差异方案；差异必须先由用户批准。

## 硬性边界

- 不得依赖 JSplitter、Spider Monkey Panel、JScript Panel 或其他 JavaScript UI 宿主。
- 初始目标为 Windows 10/11 x64。
- 基准运行环境为 foobar2000 2.25.10 stable。
- UI 使用原生 C++ 和 Win32；绘制优先使用 Direct2D 与 DirectWrite。
- 未经用户批准不得增加运行时依赖。
- 不得修改、删除、格式化或生成文件到参考目录 `D:\Dev\foobar2000`。
- 不得修改或部署到外部 Foobar-Reference。
- 只能部署到 `.local/foobar-dev` 或 `.local/foobar-test`。
- 日常使用的 C 盘 foobar2000 安装严格不在项目范围内。
- 不得把 foobar2000 二进制或第三方闭源组件提交进仓库。SDK 和第三方源码仅在许可证及仓库策略明确允许时保留，并必须记录来源与许可。

## 参考项目规则

- `D:\Dev\foobar2000` 是只读的行为参考，可以检查文件并为观察行为而启动，但绝不能修改、清理、部署或作为开发目标。
- Foobox 脚本只能用于提取可观察行为、文案、状态和交互线索；不得照搬代码，也不得把脚本内部结构当作本项目架构。
- 从脚本推断但尚未在界面实际观察的结论必须标记为“待观察”，不能写成已确认事实。
- 规格不明确时先记录问题和观察方法，不复制脚本中的偶然行为。
- 每次参考观察都应在对应任务中记录参考版本、观察入口、操作步骤、结果和必要截图。

## 规格与任务规则

- 用户可见功能必须先有对应的 `specs/` 中文规格，并经用户核对后才能进入实现。
- `specs/000-product-spec.md` 描述产品边界和整体行为；`specs/001-ui-ux-goals.md` 描述 UI/UX 目标；后续模块使用独立编号规格。
- `tasks/README.md` 是跨会话的唯一任务入口，必须能看出当前任务、状态、下一步以及用户需要检查的内容。
- 每项任务使用独立目录，目录内至少包含任务说明、步骤清单、中文程序逻辑、验证记录和决策记录；尚未产生的内容可以在任务说明中列为待创建产物，但不得用虚假空文件表示完成。
- 任务状态统一为：待讨论、规格核对中、可实现、实现中、实现完成待验收、已验收、受阻。
- 未经用户明确核对，规格任务不得标记为“可实现”；未经用户明确验收，实现任务不得标记为“已验收”。
- 规格、代码与验收行为必须保持同步；行为变化时同一任务内先更新规格。

## 工程规则

- 明确区分 UI、foobar 服务适配、状态模型和持久化。
- foobar 回调进入 UI 前必须处理线程归属和对象生命周期。
- 绘制代码不得执行阻塞式 I/O。
- SDK/foobar 对象不得在其允许的生命周期之外保存。
- Direct2D、DirectWrite、GDI 和 Win32 资源必须使用 RAII 管理。
- 必须正确处理窗口销毁、组件卸载、设备丢失、播放停止以及异步结果晚到。
- 新增设置必须具有稳定的 GUID、明确的数据类型、合理默认值、兼容旧值的策略和恢复默认方式。
- 必须支持运行时 DPI 变化，不假定固定像素密度。
- 用户操作必须覆盖空列表、无封面、无媒体库、播放停止、文件失效、权限不足和服务不可用等适用状态。
- Release 中不得留下会暴露隐私、路径或大量刷屏的调试日志。

## 必须执行的工作流

1. 严格验证编码并完整阅读本文件、任务索引、当前任务和相关规格。
2. 检查 Git 状态，识别并保护用户已有修改。
3. 用中文说明本轮范围，并把跨文件或跨子系统工作写入当前任务计划。
4. 只读检查参考资料，将“已观察”“由文件证据推断”“待确认”分开记录。
5. 先修改规格和中文程序逻辑，等待用户核对需要核对的设计决定。
6. 实现一个规格完整、可独立构建和验收的纵向功能单元。
7. 构建适用的 x64 Debug 与 Release 配置，消除新增警告或记录不可消除原因。
8. 运行相关自动测试，并确认未破坏已验收功能。
9. 仅部署到独立开发实例，提供可重复、逐步、面向非程序员的人工验收步骤。
10. 把证据写回任务文档；实现完成后等待用户验收。
11. 用户要求提交时，一个已验收功能对应一个提交；不提交无关文件，不覆盖用户已有修改。

## “实现完成”的定义

- 规格要求的正常流程、边界状态、失败处理、设置和恢复行为全部实现，不存在占位或假数据。
- 要求的构建配置编译成功，新增编译警告已消除或有批准的记录。
- 相关自动测试通过，已有自动测试和已验收功能未被破坏。
- 实现、中文程序逻辑和规格保持同步。
- 资源、回调、线程、异步任务和窗口生命周期经过检查。
- 已在独立开发实例中部署，并提供用户可以照做的人工验收步骤。
- 当前任务文档列出改动文件、检查结果、已知差异和剩余风险。

## “已验收”的定义

- “实现完成”的条件全部满足。
- 用户在指定开发实例中完成人工验收。
- 用户明确确认可见行为符合对应规格。
- 任务索引和对应规格更新为“已验收”。
